## What surprised me / what I learned
- **`read` line-mode silently swallows a poke.** A wake poke is one newline-less byte. A line
  oriented `read -u 4 x` consumes that byte but keeps blocking for a `\n` that never arrives — and
  because the wake FIFO is held open read-write (`exec 4<>`) it never EOFs either. So the wake is
  *received and then ignored*, the loop hangs until the poll floor, and you'd see mysterious
  latency, not an error. Fix: `read -N 1` (read exactly one byte, delimiter-agnostic). This is the
  kind of bug that would have masqueraded as "the wake doesn't help much" and quietly undermined the
  whole latency story. Catching it in a 30-line primitive probe, not in the 5000-trial harness, was
  the highest-leverage 10 minutes of the session.
- **A non-exiting signal trap can't break a blocked `read`.** `trap 'running=0' TERM` looks correct
  but bash *restarts* the interrupted `read` after the handler returns, so the loop never re-checks
  `running` and `wait` hangs forever. The small-N harness run timed out at exit 124; I reproduced
  the bug and the fix (`trap 'exit 0'`) in isolation before touching the scheduler. Diagnosis over
  guessing paid off — my first instinct ("the lockstep is deadlocked") was wrong; the lockstep was
  fine, teardown was the culprit.
- **Command substitution + a backgrounded long-lived child = silent hang.** `pid="$(start_sched)"`
  hung because the scheduler inherited the `$(...)` stdout pipe and held it open. Redirecting the
  child's fds to a log fixed it. Classic, but easy to forget under time pressure.
- **Deterministic injection is both simpler and stronger than probabilistic.** Rather than fire
  publishes with random timing and *hope* they land in the `[MIN, read]` window, the test hook
  freezes the loop *in* the window and the harness injects while it's frozen. 100% in-window, and
  the lockstep barrier ("see the next pause signal") tells the harness precisely when the fire
  completed — no sleeps, no flakiness, no sampling error.

## M1 — deferred crash recovery (same session)

Built `tests/crash_recovery.sh` (RED first) then added a startup outbox-recovery sweep to
`src/sched.sh` (GREEN). Insight: **deferred re-arm was already true by construction** — `scan_min`
reads `deferred/` from disk every iteration, so surviving pending files re-arm with zero new code.
What M1 genuinely adds is **outbox recovery**: the `mv` commit point moves a record into `outbox/`,
and a crash strictly between that `mv` and the delivery+`rm` would otherwise strand the record
forever. The sweep re-delivers any `outbox/` survivor on startup, realizing
at-most-once-modulo-crash. The must-fail control is `SCHED_NO_RECOVER=1` (one knob): it skips the
sweep and the test asserts the stranded file is LOST — proving the recovery code is what saves it.

Two more harness bugs, both diagnosed not guessed:
- `fired_count` used `grep -c ... || echo 0`; `grep -c` prints `0` *and* exits 1 on no-match, so the
  `||` appended a second `0` → `"0\n0"`, breaking every integer comparison (the `[: 0\n0:` spam).
  Fixed to capture once and default `${n:-0}`.
- The `${2:-} VAR=val cmd` env-prefix idiom mis-parses under `set -u` when `$2` is empty: `VAR=val`
  gets treated as a *command* (`SCHED_IDLE_POLL_MS=200: command not found`), so the scheduler never
  started and the harness blocked forever on a poke with no reader. Fixed with explicit `env`.

Confirmed NO M0 regression: smoke green, chaos still missed=0/dup=0 with all controls failing. The
recovery sweep is a no-op when `outbox/` is empty (the steady state).

## M2 — acceptor + SUB handler + per-subscriber FIFO (same session)

The broker proper. Studied terminalphone's real handler (terminalphone.sh:1514-1596) — the drainer
(:1540), keepalive `exec 3>` (:1546), `[ -p ]` fan-out with the `> $f &` leak (:1567-1572), and
cleanup-at-script-end (:1592, NOT a trap — matching the docs' honesty note). Before writing a line
of broker, I probed the two riskiest socat semantics (same discipline that caught the read -N1 bug):
  - `socat UNIX-LISTEN/TCP-LISTEN ,fork EXEC:handler` → each connection forks its own handler with
    the socket on stdin/stdout; a backgrounded drainer writing stdout reaches the client; concurrent
    connections fork independent handlers. ✓
  - client disconnect → handler stdin EOF → EXIT trap unlinks the FIFO; `pkill -P` reaps handlers. ✓

Then TDD: `tests/sub_lifecycle.sh` RED-first (S1 SUB registers FIFO, S2 forget-on-death, S3
concurrent isolation, S4 TCP, S2' must-fail no-trap control), then `src/shellmux` to green it.

Two real bugs, both diagnosed not guessed:
- **socat `EXEC:` does not use a shell** — it execvp's the first token. My `SHELLMUX_NO_TRAP=0 bash
  ... _handle` prefix became the literal program name (`execvp("SHELLMUX_NO_TRAP=0", ...): No such
  file`). Fix: export the var into socat's environment; socat propagates it to the forked child.
- **The `EXIT` trap saw `INFIFO` as unbound** because it was `local` to `_handle` while the trap
  fires in top-level scope; under `set -u` the trap aborted before the `rm`, so the FIFO stranded —
  indistinguishable from the no-trap control. Fix: make `INFIFO` a script-global (one connection per
  handler process, so it's safe). This one was subtle: S2 and S2' BOTH failed identically, which is
  exactly the "negative control can't be distinguished from the real path" smell the must-fail
  discipline is designed to surface.

NO M0/M1 regression. `src/shellmux` is 172 lines (PUB still a stub; M3 adds fan-out + ring drainer).
