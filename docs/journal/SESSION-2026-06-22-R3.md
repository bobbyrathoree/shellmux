### 4. Fixed the two real (off-axis) bugs, test-first, each with a must-fail control

**Bug A — corrupt deferred filename halts the scheduler (global-liveness DoS).**
- `deferred/<non-numeric>.<seq>` (publisher crashed mid-write, or a raw producer writing into the
  filesystem-native state dir) poisoned `next=MIN(run_at)`: `scan_min` assigned the bareword into
  `$min`, the loop's `to_ms=$(( next - now ))` hit `set -u` → `unbound variable` → scheduler DIED.
  One bad file blocked delivery of every other pending well-formed record, deterministic across
  restarts. NOT reachable through the validated broker (`valid_deadline` rejects non-numeric
  `--at`/`--delay` before staging; `run_at_ms` is integer arithmetic).
- Fix: `is_numeric()` guards `scan_min` + `fire_due` to SKIP a non-numeric prefix — a corrupt file
  costs one scan-skip, never a halt (same posture as "a spurious/dropped wake costs one scan, never a
  missed message"). One-knob must-fail control `SCHED_NO_SKIP_CORRUPT=1`.
- `tests/corrupt_deferred.sh` (C1 + C1'); RED confirmed the crash, GREEN 2/2. Commit `a89f61f`.

**Bug B — torn/concatenated frames under concurrent >PIPE_BUF publishers.**
- `fanout()` wrote each frame with a fresh `open()`+`write()`; a FIFO write is atomic only up to
  PIPE_BUF (4096B). Two+ concurrent socat-forked publishers fanning records >PIPE_BUF to the SAME sub
  interleaved their bytes, and a torn/concatenated frame whose size matched the length prefix slipped
  past the drainer's short-read guard and reached a HEALTHY subscriber corrupted. Reproduced
  independently: 6 pubs × 6000B → corrupt=2 across 3 trials. Contradicted the spec's length-prefix
  "not concatenated" guarantee and flood_wedged F4 (which only ever covered 1 publisher, <PIPE_BUF).
- Fix: serialize each per-FIFO fan-out write under a **per-subscriber** `flock` (`.wlock_<pid>`) so a
  >PIPE_BUF frame is atomic w.r.t. other publishers. Per-sub (NOT global) → writes to different subs
  still run in parallel and a wedged peer never blocks a healthy one (M3 isolation preserved). Stays
  `timeout`-bounded → backpressure contract unchanged. `_reap` now sweeps stale `.wlock_<pid>` like
  `drops_<pid>`; `introspection.sh` I5 asserts it. One-knob must-fail control `SHELLMUX_NO_WLOCK=1`.
- `tests/concurrent_frames.sh` (G1 + G1'); RED corrupt=2, GREEN locked=360/0 corrupt. Commit `896b1b9`.

## Lessons
- **Subshell vs fresh-exec RANDOM is a real trap.** An adversarial agent can "reproduce" a collision
  that the production path never exhibits, because `( )` inherits the seed and `execvp` reseeds.
  Always model the actual spawn mechanism (socat `EXEC:` = fresh bash) before believing a repro.
- **The chaos harness can only prove what it reaches.** Both real bugs lived in paths it structurally
  cannot exercise (corrupt on-disk state; concurrent >PIPE_BUF fan-out). A finder fanned across the
  *unreached* paths is the right complement to the in-window race proof.
- **Off-axis ≠ ignore.** Neither bug threatened missed=0/dup=0, but both contradicted a *stated*
  guarantee (the spec's "not concatenated"; global liveness). The honesty ethos says fix them or
  document them — fixing was cheap and made the docs true.

## Real-hardware measurement on AWS Graviton (the Pi follow-up, closed)
The user asked whether the "measure on a real $5 Pi" follow-up could be done in their AWS account
instead. Yes — Graviton is ARM64 bare metal with the real dep set, far more faithful than Docker-on-
macOS, while being honest that Graviton cores are faster than a Pi's (so throughput is an upper
bound; the RAM-bound ceiling transfers directly).

Automated end-to-end via the AWS CLI (default profile, us-west-2): tar'd src+tests → S3, EC2
user-data bootstrap (dnf install real deps, run_all.sh N=5000 + bench.sh + a subscriber-ceiling
probe), results back via S3/SSM. Launched **t4g.small (2 GB)** and **t4g.nano (0.5 GB ≈ Pi RAM)** on
AL2023, then **tore down everything** (instances, bucket, IAM role/profile, SG — verified zero
residual).

Findings (`docs/evidence/R3-aws-graviton.md` + `R3-aws-graviton-t4g.small.txt`):
- **Correctness is hardware-independent — demonstrated.** chaos N=5000 = `missed=0 dup=0` at 0.00%
  idle CPU on BOTH boxes, including the 0.5 GB box *under active OOM-killer pressure*. Kills the
  "never left the container" criticism.
- **Throughput is fork-bound: ~52-60 msg/s/sub on 2 vCPU** — ~20× below the 14-core container's
  ~1360. One `timeout bash -c` per record per sub + 2 cores = the honest small-hardware number.
- **Subscriber ceiling is RAM-bound: ~2.6 MB/sub, ~150-175 subs on the 0.5 GB box** before the OOM
  killer engaged (dmesg confirmed) — validates and sharpens the "~100-150 on a 512 MB Pi" estimate.
- **Graceful degradation:** the nano's `flood_wedged` went 3/1 (one healthy sub got 999/1000) only
  because the kernel was OOM-killing processes mid-run — best-effort fan-out degrading 0.1%, not a
  crash/leak, exactly as the honesty docs promise. The 2 GB box was 10/10.

Lesson: a constrained cloud box is a legitimate, honest stand-in for "real small hardware" as long
as you label the architecture and don't claim it IS the Pi. The OOM behavior on the nano was a bonus
— it exercised the graceful-degradation contract for free.
